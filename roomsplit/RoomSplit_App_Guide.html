🏠 What is RoomSplit?

RoomSplit is a full-stack web application designed for roommates in India to manage shared rent and household expenses with zero friction. An admin creates a group, invites roommates, defines monthly rent cycles, and collects payments entirely through UPI — no cash, no spreadsheets, no chasing people on WhatsApp.

Key promise: Every rupee is tracked, every payment is audited, and every roommate always knows exactly what they owe and what has been paid.

🔐 How Authentication Works

RoomSplit uses phone number + OTP — no passwords are stored.

1
User enters phone number 10-digit Indian mobile, formatted as +91XXXXXXXXXX
2
OTP sent via Twilio (or printed to console in development) 6-digit code, expires in 10 minutes
3
User submits OTP → server issues tokens Access token (JWT, 30 min) + Refresh token (SHA-256 hash, 30 days)
4
Access token kept in memory only (Zustand) Never stored in localStorage — protects against XSS attacks
5
On token expiry → silent refresh Refresh token in HTTP-only cookie → new access token issued automatically
New users are directed to a Profile Setup page (name + UPI ID) after their first successful OTP verification before accessing the app.

👥 Groups & Members

A Group represents a shared household. One user can belong to multiple groups. The person who creates a group automatically becomes its Admin.

Creating a Group

The admin fills in:

  • Group Name — e.g. "Flat 4B Residents"
  • Rent Collection UPI ID — the UPI handle where rent payments are collected
  • Cycle Type — Monthly (fixed billing day) or Custom (admin-managed)
  • Billing Day — for monthly cycles, the day of month rent is due (1–28)

Inviting Members

Admin goes to Members → Invite Code. Roommates enter this code on the Join Group page. Invite codes expire after 7 days; admin can refresh them anytime.

Member Management

  • Admin can remove members (except the group creator)
  • Admin can assign roles (Admin / Co-Admin / Member or custom)
  • Admin can suspend members — they remain in records but cannot participate
  • Admin can delete the group — soft delete, all data is preserved in the audit log

🔄 Rent Cycles

A Rent Cycle is a billing period (e.g. "March 2025"). The group is permanent — you create a new cycle each month, not a new group.

Cycle Lifecycle

Draft → activate → Active → close → Closed
1
Admin creates a Cycle (Draft) Sets label, period start/end, total rent amount (₹), and due date. Nothing is charged yet.
2
Admin activates the Cycle System divides total rent equally across all active members (remainder paise distributed to first members). A Payment record is created for each member and a "Rent Due" notification is sent.
3
Members pay their share Each member sees their payment, scans the UPI QR code or uses the deep link, pays, then marks it as paid with their UTR (bank reference number).
4
Admin confirms payments Admin verifies UTR and confirms. Wallet balance and ledger are updated. If auto-confirm is ON, this step is skipped.
5
Admin closes the Cycle Once the period ends, admin closes the cycle. Historical records remain accessible.
Equal split only (Phase 1): Custom per-member splits are planned for Phase 2. Currently the total is divided equally, with leftover paise distributed one paise at a time to the first N members to ensure the total always adds up exactly.

₹ Payment Flow

Each Payment record goes through the following states:

Pending → member marks paid → Marked Paid → admin confirms → Confirmed
or → Rejected (member must pay again)
Step Who Action Result
1 System Cycle activated Payment created, UPI QR generated
2 Member Scans QR / taps UPI link, pays via bank app Money transferred (outside app)
3 Member Enters UTR, clicks "Mark as Paid" Status → Marked Paid
4a Admin Verifies UTR in bank, clicks Confirm Status → Confirmed, wallet credited
4b Admin UTR doesn't match, clicks Reject Status → Rejected, member notified
Auto System auto_confirm_payments = ON Step 4 is skipped; confirmed automatically

💳 Wallet & Ledger

Each group has a Group Wallet that accumulates confirmed rent payments. The Ledger is an immutable list of every credit and debit transaction.

  • Credit — when a member's payment is confirmed, the amount is credited to the group wallet
  • Debit — when a withdrawal is approved, the amount is debited from the wallet
  • All amounts are stored as integer paise (1 ₹ = 100 paise) — never as floating-point numbers
  • Concurrent wallet updates use SELECT FOR UPDATE to prevent race conditions

Withdrawals

Admin can request a withdrawal from the wallet (e.g. to pay the landlord). Another admin or co-admin approves it. Once approved, the wallet balance is reduced and a debit ledger entry is created.

🔔 Notifications & Background Tasks

In-App Notifications

Notifications are shown in a slide-in drawer (top-right bell icon). They are triggered for:

  • rent_due — when admin activates a cycle
  • payment_confirmed — when admin confirms your payment
  • payment_rejected — when admin rejects your payment

🏗️ Key Architecture Decisions

Money stored as paise (integer)
All currency values are BigInteger paise (1 ₹ = 100 paise). Floats are never used internally. Conversion to ₹ happens only at the API boundary. This eliminates floating-point rounding errors on INR calculations.
Payment state machine
A VALID_TRANSITIONS dict enforces legal state changes (e.g. pending → marked_paid is valid; confirmed → pending is not). This prevents data corruption from concurrent or out-of-order API calls.
Wallet SELECT FOR UPDATE
Wallet balance updates use row-level locking to prevent two simultaneous confirmations from both crediting the same amount, which would create a double-count bug.
Audit log PostgreSQL RULE
A DB-level RULE blocks UPDATE and DELETE on the audit_logs table. Even if application code has a bug, the audit trail cannot be tampered with — it is append-only at the database level.
Refresh tokens as SHA-256 hashes
Raw refresh tokens are never stored. Only their SHA-256 hash is saved. If the sessions table is leaked, attackers cannot use the hashed values to authenticate.
Access tokens in memory only
JWT access tokens are stored in Zustand (in-memory), never in localStorage or cookies. This limits the XSS attack surface — a malicious script cannot steal the token by reading storage APIs.
RBAC resolved at query time
Permissions are always fetched live from the DB on each request. There is no permission cache to invalidate. Changing a member's role takes effect immediately on the next API call.

🚀 Setup & Deployment

git clone https://github.com/your-username/roomsplit.git && cd roomsplit
cp .env.example .env

# 2. Start everything
docker-compose up --build

# App is live at http://localhost:8080
# API docs at http://localhost:8000/docs
